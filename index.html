計測データの見方

画面上部に表示されている各データの意味は以下の通りです。

• 品質スコア（アルファベット）: 総合的な測位精度のランクです。S（最高精度・5m未満）、A（高精度・15m未満）、B（実用・50m未満）、C（低精度）、D（参考値）の5段階で評価されます。
• ◎ ±XXm: 現在の座標に含まれると推定される誤差半径（メートル）です。
• ⏱ XX.Xs: 初回の測位開始から現在までに経過した時間です。
• 📡 Xサンプル: ここまでの計算に利用された観測データの数です。
• ✦ KF +XX%: カルマンフィルタによる補正によって、デバイスからの生データよりもどれだけ精度が向上したか（ノイズが除去されたか）の割合を示します。
• 精度推移グラフ: 観測開始から現在にいたるまでの誤差半径（±XXm）の推移を視覚化したものです。右に行くほど値が小さくなり、精度が「収束」していく様子がわかります。

idokeidoの設計思想

「idokeido（緯度経度）」は、モダンブラウザが提供する W3C Geolocation API のポテンシャルを極限まで引き出し、最高精度の座標を最も直感的かつ美しく取得・表示するための実験的Webアプリケーションです。

単なる現在地の取得にとどまらず、デバイスのGNSSやWi-Fi測位から得られる生のセンサーデータ（Raw Data）に内在するノイズを数学的アプローチで除去し、信頼に足る強固な座標データへと「収束」していく過程を可視化しています。GPSの制約やOSの抽象化レイヤーを意識することで、位置情報技術の深淵に触れることができます。

2次元カルマンフィルタによる平滑化

GPSから得られる座標には、大気圏の遅延や建物によるマルチパス（反射）による誤差が必ず含まれます。
本システムでは、JavaScript上でゼロからインライン実装された Kalman Filter（カルマンフィルタ） をリアルタイムに適用しています。

状態ベクトルに「座標（x,y）」と「速度（vx,vy）」の4次元を持たせ、毎回の測定値とその測定精度（Accuracy）を観測ノイズとして動的モデルに適用します。これにより、急激な座標の「飛び」や「ジグザグ移動」を抑制し、実際の真の移動に沿った極めて滑らかな座標（上部情報チップでの Δ補正：フィルタにより生データから空間的にどれだけ座標が補正されたかを示す距離）を導き出します。複雑な行列演算をブラウザ内で自己完結させている点が技術的な独自性です。

計算の数学的基盤

本システムは、等速直線運動を前提とした物理モデルを採用しています。
状態ベクトルを x = [px, py, vx, vy]T と定義します。ここでの計算は、デバイス側から新しい位置情報（OSやハードウェアに依存した数秒に1度の不定期なデータ）が受信されるたびに実行されます。

まず、F を測定間隔（Δt）に応じた状態遷移モデル、P を計算された推定誤差（不確実性）を表す共分散行列、Q を予期しない加速度などを示すプロセスノイズと定義します。これらを用いて、前回から現在までの座標の「予測」を行います。

予測ステップ (Predict):
xk|k-1 = F · xk-1|k-1
Pk|k-1 = F · Pk-1|k-1 · FT + Q

次に、zk をセンサから新しく得られた観測値（局所座標）、R をOSから提供されるAccuracyに基づく観測ノイズの共分散行列と定義します。
この時、K（カルマンゲイン）と呼ばれる重み係数を計算し、「予測した位置」と「センサが示した位置」のどちらをどれだけ信用するかを理論的に決定し、「更新（補正）」を行います。

更新ステップ (Update):
K = Pk|k-1 · HT · (H · Pk|k-1 · HT + R)-1
xk|k = xk|k-1 + K · (zk - H · xk|k-1)
Pk|k = (I - K · H) · Pk|k-1

精度収束ロジックと除外処理

初回の測位データは誤差が数千メートルに及ぶこともあり、システムは直ちに結果を確定しません。内部では watchPosition() を利用して継続的な観測を行い、カルマンフィルタによる推定誤差が 10m 以下になるまで収束プロセスを継続します。ただし、いつまでも結果が出ない状態を防ぐため、観測開始から 10秒 経過した時点での最良値を最終結果として採用して確定させるよう設計されています。

また、時速300kmを超えるような物理的に不可能な移動（座標ジャンプ現象）が検出された場合は、異常値として即座に破棄する堅牢なロジックを搭載しています。これにより、ビルの谷間など厳しいマルチパス環境下での突発的なノイズを効果的に排除します。

プライバシー保護と制約

本番環境において、測位からフィルタリングまでの計算処理はすべてクライアント（あなたのブラウザ）内で完全に完結して実行されます。位置情報がいかなる外部サーバーにも送信されることは一切ありません。

※iOS環境においては、OSのプライバシー設定において「正確な位置情報」がオフになっている場合、OS側で意図的に大幅な誤差が加えられます。システムはこの特有の誤差分布を検知し、品質スコアを低下させて警告を表示します。これはプライバシー保護機能と利便性のトレードオフを可視化する試みでもあります。

Phase 3: 外部GNSS連携

この機能は極めて試験的（Experimental）であり、通信の切断やパースエラーなど、圧倒的に不安定な挙動を示す可能性があります。実用的なナビゲーション用途ではなく、技術実証を主な目的としています。

Web Serial API または Web Bluetooth API を使用して、u-blox ZED-F9P 等の外部GNSSレシーバーに直接接続します。ブラウザの標準的な Geolocation API による抽象化を完全にバイパスし、生の NMEA センテンスを JavaScript で直接解析することで、以下のような専門的な空間データへ直接アクセスします。

• 可視衛星・使用衛星数（$GPGSV/$GLGSV）: 上空のGNSSコンステレーション状況
• HDOP/VDOP/PDOP（精度低下率）: 衛星配置による幾何学的な測位誤差の増幅指標
• Fix品質（単独/DGPS/RTK Float/RTK Fixed）: 搬送波位相測位の解のステータス
• 測位誤差の標準偏差（$GPGST）: mm/cmクラスの厳密な精度推定値

良好な観測条件下で RTK Fixed 状態を引き出すことができれば、ブラウザ上で 1〜3cm 程度の驚異的な精度を実現することが可能です。
※ Chrome系ブラウザ（Chrome, Edge, Opera）でのみ動作します。Safari / Firefox等では対象APIがサポートされていないためご利用いただけません。
